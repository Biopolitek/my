Техническое задание для модуля ИИ-кодера (Asset-to-Code Transformation)
Объект задачи: Автоматизированная сборка веб-интерфейса на основе экстрагированных данных.
1. Анализ входных данных (Input Data)
Твоим источником истины является папка экспорта со следующей структурой:
/assets/images/: Все уникальные растровые элементы (PNG).
/assets/vectors/: Геометрические примитивы и иконки (SVG).
/data/styles.css: Технический манифест координат. Каждому файлу в папке images соответствует CSS-класс с тем же именем.
/data/index_draft.html: Инструкция по иерархии DOM и текстовому наполнению.
map_assets.txt: Спецификация шрифтов и системных требований.
2. Инструкции по визуальной сборке (Pixel-Perfect Logic)
Позиционирование: Используй данные из styles.css. Каждый элемент должен быть спозиционирован через position: absolute относительно родительского контейнера с position: relative. Это гарантирует 100% совпадение с макетом.
Z-Index: Строго соблюдай z-index, указанный в CSS. Он соответствует иерархии слоев в исходном шаблоне.
Векторная графика: Объекты из /assets/vectors/ предпочтительно вставлять инлайново (inline SVG) для обеспечения четкости при масштабировании и возможности изменения цвета через CSS (fill: currentColor).
3. Работа с типографикой
Используй список из map_assets.txt.
Для всех шрифтов со статусом MISSING, сгенерируй блок @font-face или найди соответствующий импорт из Google Fonts.
Параметры font-size, line-height и letter-spacing (если они извлечены в CSS) являются приоритетными перед стандартными браузерными стилями.
4. Трансформация в функциональный код
Семантика: На основе имен классов (например, btn_submit, header_logo, nav_link) преобразуй стандартные <div> из index_draft.html в семантические теги HTML5 (<button>, <header>, <nav>, <a>).
Оптимизация: Если несколько изображений имеют идентичные размеры и визуально являются частями одной группы (например, иконки соцсетей), сгруппируй их в один Flexbox-контейнер, сохраняя общие внешние координаты группы.
Форматы: При финальной сборке замени ссылки на изображения с .png на .webp, если браузер пользователя поддерживает этот формат (для ускорения загрузки), но сохрани оригинальные PNG для Safari старых версий.
5. Аналитический отчет (CMS Choice)
Проанализируй итоговую сложность DOM-дерева:
Если количество абсолютных слоев > 100 и они не образуют четких сеток — рекомендуй Static Site Generation (SSG) на базе Next.js или Astro.
Если структура повторяющаяся (блочная) — подготовь структуру для интеграции в Headless CMS (Strapi/Directus).
Почему это сработает (Аналитика для оператора)
Мы предоставили ИИ не просто картинки, а числовую модель макета:
ИИ не нужно "угадывать" расстояние между кнопками — оно прописано в CSS.
ИИ не нужно распознавать текст — он уже выгружен в HTML-черновик.
ИИ не нужно искать шрифты — у него есть список и готовые ссылки на поиск.
Результат: Сборка будет выполнена автономно, а финальное сравнение с эталоном из /data/reference/ подтвердит качество Pixel-Perfect.








сборка кода На входе предоставлен пакет ресурсов, экспортированный из графического шаблона (2026 год). Твоя задача — собрать функциональный сайт с попиксельной точностью (Pixel-Perfect).
Состав пакета:
/assets/images/: Все растровые элементы в формате PNG.
/assets/vectors/: Векторные слои и страницы в формате SVG. Используй их как background-image или инлайново для сохранения четкости.
/data/styles.css: Содержит точные координаты (left, top), размеры и z-index для каждого элемента. Внимание: Используй эти данные как основу для CSS-классов.
/data/index_draft.html: Предварительная структура DOM с именами классов, соответствующими CSS.
map_assets.txt: Список шрифтов. Если статус шрифта MISSING, ты должен найти подходящий аналог на Google Fonts или предложить @font-face подключение.
Требования к сборке:
Layout: Используй position: absolute внутри контейнера relative для первичной сборки 1:1. Затем проанализируй группировку элементов и попытайся преобразовать их в Flexbox или CSS Grid, если это не нарушит визуальную точность.
Оптимизация: Все изображения в HTML должны иметь alt атрибуты, взятые из имен классов.
CMS-Адаптация: На основе структуры (определяемой по index_draft.html), предложи наиболее подходящую CMS. Если структура сложная и нелинейная — используй статический генератор (Next.js/Astro). Если блочная — WordPress.
Шрифты: Обеспечь корректное отображение текста. Если в CSS указан шрифт, которого нет в системе, замени его на максимально похожий веб-безопасный шрифт из той же семьи (serif/sans-serif).
